# GPT Pro 검토 요청: 작업중지권 진성 과판정 개선

아래 내용은 작업중지권 `진성/가성` 자동 판정 프로젝트의 현재 상황 요약입니다.
목표는 프롬프트를 더 길게 만드는 것이 아니라, 최근 데이터에서 반복되는 `가성 -> 진성` 오판을 줄여 `TP(진성 정밀도)`를 높이는 것입니다.

## 1. 프로젝트 목적

작업중지권 등록 건을 생성형 AI로 `진성` 또는 `가성`으로 분류합니다.

- 진성: 작업 중 또는 작업 개시 직전, 작업자가 실제 중대재해 위험에 노출되어 작업 또는 공정 중단이 필요한 경우
- 가성: 작업 전 일반점검, 행정/절차, 단순 시정, 계획정비, 협력회사 자체 조치 등 긴급성과 필요성이 부족한 경우

운영에서는 AI가 예측한 결과를 DB에 적재하고 담당자가 참고합니다.
따라서 단순 accuracy뿐 아니라, AI가 `진성`이라고 한 건이 실제로도 진성인지가 매우 중요합니다.

## 2. 현재 판정 구조

현재 로직은 대략 다음과 같습니다.

1. HTML 컬럼에서 텍스트와 이미지(base64)를 추출합니다.
2. LLM에게 이미지와 텍스트를 함께 전달하여 evidence를 풍부하게 만듭니다.
3. `current_policy.md`에 따라 진성/가성을 판단합니다.
4. 현재 운영 후보 로직은 `true_confirm`입니다.
   - 같은 건을 2회 판단합니다.
   - 2회 모두 `진성`일 때만 최종 `진성`.
   - 하나라도 `가성`이면 최종 `가성`.

이 방식은 기존 단일 판정보다 진성 남발을 줄이기 위한 장치입니다.

## 3. 최근 데이터 기준 문제

과거 전체 데이터(2025년 포함)에서는 그럭저럭 좋아 보였지만, 최근 2026년 데이터 약 3,142건 기준으로 성능을 다시 보니 문제가 명확해졌습니다.

최근 데이터 기준 대표 성능:

- ACC: 약 0.86
- TR(True Recall, 실제 진성을 진성으로 맞춘 비율): 약 0.82~0.83
- TP(True Precision, AI가 진성이라고 한 것 중 실제 진성 비율): 약 0.60~0.61
- FR(False Recall, 실제 가성을 가성으로 맞춘 비율): 약 0.86~0.87
- FP(False Precision, AI가 가성이라고 한 것 중 실제 가성 비율): 약 0.95~0.96
- FN(True -> False): 104건
- FP(False -> True): 327건

핵심 해석:

- 실제 진성은 꽤 잘 잡고 있음(TR 높음).
- 하지만 AI가 진성을 너무 많이 남발함(TP 낮음).
- 운영 관점에서는 `AI 진성 비율`이 높아져 담당자가 "AI가 너무 진성을 많이 준다"고 느낄 위험이 큼.
- 현재 가장 중요한 목표는 `TP를 올리는 것`, 즉 `가성 -> 진성` 오판을 줄이는 것.

## 4. 목표 지표

현실적인 목표:

- ACC: 0.88 이상, 가능하면 0.90 이상
- TP: 최소 0.70 이상
- TR: 가능하면 0.70 이상 유지

현재는 TR이 높은 대신 TP가 낮으므로, 약간의 TR 하락을 감수하더라도 TP를 개선하는 방향이 필요합니다.

## 5. 최근 FP(False -> True) 주요 군집

최근 데이터에서 가장 큰 문제는 `pipe/support stepping` 계열입니다.
즉, 배관/서포트/발판/협의/공간협소/DS조치건/협력회사 자체 개선완료 같은 표현이 나오면 AI가 과하게 진성으로 끌려가는 경향이 있습니다.

대표 FP 군집:

```text
pipe/support stepping | 협력회사 자체 개선완료 | 62
terms: 해당없음, 확인, 밟, 협력회사, 진성, 배관, 자체, 개선완료, 협의, 발판, 불명확, 사다리

pipe/support stepping | 추락/낙하 | 58
terms: 밟, 협의, 해당없음, 추락, 발판, 배관, 공간, 진성, 확인, 사다리, 밟음, 아니오

pipe/support stepping | 삼성전자 임직원 요청완료(DS조치건) | 40
terms: 배관, 해당없음, 확인, 진성, 불명확, 아니오, 설비, 협의, 공간, 발판, 간섭, 협소

space/path/equipment interference | 협력회사 자체 개선완료 | 23
terms: 해당없음, 확인, 불명확, 진성, 아니오, 자체, 협력회사, 개선완료, 사다리, 가스, 구체적, 작업중지

pipe/support stepping | 충돌 | 20
terms: 배관, 공간, 해당없음, 충돌, 진성, 협소, 협의, 불명확, 아니오, 확인, 간섭, 설비

contractor self-correction | 협력회사 자체 개선완료 | 15
terms: 해당없음, 불명확, 진성, 자체, 확인, 개선완료, 아니오, 협력회사, 협의, 상부, 교체, 고소
```

중요한 관찰:

- FP 군집에 `해당없음`, `불명확`, `아니오`가 매우 자주 등장합니다.
- evidence 단계에서는 실제 밟음 강제성이나 작업자 노출이 불명확하다고 잡히는 것 같습니다.
- 그런데 최종 판정 프롬프트가 `배관`, `협의`, `손들기`, `DS조치건`, `협력회사 자체 개선완료`, `추락/낙하` 같은 단어에 끌려 여전히 진성으로 판단하는 듯합니다.

## 6. 최근 LLM 분석 리포트 요약

최근 오류군집을 사내 LLM에 분석시킨 결과는 다음과 같습니다.

### root cause

- 모델이 `작업 개시 직전`을 과도하게 확장하여 단순 사전협의나 계획 단계의 안전기준 이탈 승인 요청까지 진성으로 분류함.
- `우려`, `불량`, `미흡` 등 추상적 위험 서술만 있어도 진성으로 과대평가함.
- `협력회사 자체 개선완료`, `DS 조치건` 태그가 있는 사례에서 실제로는 행정적 완료/자체조치 맥락인데도 상세 내용의 위험어만 보고 진성화함.
- 작업 전 발견된 정적 결함(배관 꺾임, 스티커 미부착 등)이 즉시 시정 가능함에도 잠재적 위험 때문에 진성으로 분류됨.
- 미상 액체/가스감지기 알람 사례에서 사후 무해 확인보다 작업중지 당시 불확실성만 강하게 반영되어 진성 편향이 생김.

### high value policy change 후보

- `작업 개시 직전`은 실제 작업 착수 직전으로 한정하고, 단순 작업방법 승인 사전협의 단계는 제외.
- 단순 시정 가능 범위에 단순 도구 교체, 단순 위치 변경, 스티커/서류 보완 등을 명확히 포함.
- `협력회사 자체 개선완료`, `DS 조치건`은 상세 내용에 작업 중 임박한 중대재해 위험이 명확하지 않으면 원칙적으로 가성으로 볼 수 있는 특례 필요.
- 추상적 위험어만으로는 진성 금지.
- 미상 물질은 무조건 진성 처리하면 FP 증가 위험이 있음.

## 7. 현재 프롬프트에서 의심되는 문제

아래 문장들이 최근 데이터에서 진성 과판정을 유발하는 것으로 의심됩니다.

### 7.1 회사 지정 안전도구/작업방법 승인 특례가 너무 넓음

현재 취지:

```text
회사 지정 안전기준상 특정 도구·작업방법만 허용되는데,
공간 협소·설비 간섭·접근 불가 등 현장 조건 때문에 해당 기준을 그대로 준수할 수 없어
대체 도구·대체 작업방법 사용 전 삼성전자·DS·SEC·담당부서의 허락·협의·승인을 받기 위해
작업중지권을 행사한 경우는 회사 내부 판정 기준상 진성으로 판단한다.
이는 작업일이 향후 예정되어 있어도 적용한다.
```

문제 의심:

- `향후 예정되어 있어도 적용`이 너무 강해서 사전협의/이메일협의/허가요청도 진성으로 끌어올릴 수 있음.
- 담당자는 "위험해 보여서 손들기하면 진성"이라고 말하지만, 실제 라벨에서는 유사한 손들기/협의 건도 가성으로 찍는 경우가 있음.
- 따라서 이 조항은 충분조건으로 두면 위험할 수 있음.

### 7.2 pipe/support 보정 문장이 여전히 true-prime 역할을 할 수 있음

최근 추가/수정했던 문장 취지:

```text
배관·서포트·RACK·덕트 등 밟으면 안 되는 설비 관련 사례는
협의라는 단어 자체가 아니라 실제 밟음 강제성과 협의 목적을 기준으로 판단한다.
```

이 방향은 맞아 보이나, 세부 문장이 조금만 잘못되면 LLM이
`배관 + 협의 + 발판 + 추락`을 보고 진성으로 과하게 판단합니다.

### 7.3 태그 override 금지 원칙이 너무 강함

현재 정책은 대분류/중분류/태그보다 상세 내용 우선이라고 되어 있습니다.
이 원칙 자체는 맞지만, 최근 FP에서는 다음 태그가 반복적으로 보입니다.

- 협력회사 자체 개선완료
- DS 조치건
- 삼성전자 임직원 요청완료

이 태그들은 자동 가성 조건은 아니지만, 상세 내용이 불명확할 때는 오히려 `가성 쪽 약한 prior`로 써야 할 가능성이 있습니다.

### 7.4 evidence의 `해당없음/불명확/아니오`가 최종판정에서 충분히 반영되지 않음

FP 군집에 `해당없음`, `불명확`, `아니오`가 많이 보입니다.
즉 evidence가 부족하다고 말하고 있는데, 최종 프롬프트가 이를 무시하고 진성으로 올리는 듯합니다.

## 8. 담당자 확인 내용

담당자에게 짧게 물어본 결과:

- 배관 관련 건은 담당부서 허락/협의/승인 여부가 핵심 구분 기준이라고 함.
- 접액/누출 건은 작업자가 다칠 위험이 있는지가 핵심이라고 함.
- 미상 액체는 확인 전에는 강산/강알칼리인지 알 수 없으므로, 당시 작업중지는 진성으로 보는 경향이 있음.
- 작업 전에 발견했으면 작업중지 취지와 맞지 않아 가성이라는 답변도 있었음.
- 작업 당시 위험을 우선해 판단하는 게 맞다고도 함.

문제:

- 담당자 기준 자체가 완전히 일관적이지 않음.
- 특히 `손들기/협의`는 말로는 진성이라고 하지만 실제 라벨에서는 가성도 많음.
- 따라서 프롬프트는 `협의가 있었다`가 아니라 `협의 없이는 안전기준 위반 또는 물리적 위험 통제가 불가능했는지`를 더 강하게 봐야 함.

## 9. 절대 피해야 할 방향

아래 방향은 이미 여러 번 실패했습니다.

- 프롬프트를 길게 계속 덧붙이기
- `배관`, `협의`, `손들기`, `DS조치건`을 넓게 진성화하기
- 모든 작업 전 발견을 가성화하기
- 모든 미상 액체를 진성화하기
- 모든 협력회사 자체 개선완료를 가성화하기
- `~일 때만 진성` 같은 문장을 길게 넣어놓고, 그 조건에 해당하지 않으면 전부 가성처럼 읽히게 만들기
- 반대로 `~일 때만 가성` 같은 문장 때문에 그 조건 밖을 전부 진성으로 해석하게 만들기

지금 필요한 것은 긴 새 섹션이 아니라, 진성 과판정을 만드는 문장을 제거/약화하고, 불명확하면 진성으로 올리지 못하게 하는 짧고 강한 보정입니다.

### Q3. 짧은 안전한 패치 제안

프롬프트 전체를 다시 쓰지 말고, 최근 FP를 줄이기 위한 3~5문장짜리 짧은 패치를 제안해주세요.

목표:

- `pipe/support stepping`의 가성 -> 진성 오판 감소

- `해당없음/불명확/아니오` evidence가 있으면 진성 근거로 쓰지 못하게 하기
- 단, 실제 작업 중 배관/서포트 강제 밟음, 물리적 보강, 담당부서 승인 없이는 안전기준 준수 불가한 경우는 진성 유지

### Q4. 프롬프트 수정 vs 코드 후처리

예시:

```text
route가 pipe/support이고,
```

```text
   - 보호구 착용, 위치 변경, 단순 보수, 절차 준수로 즉시 통제 가능한 위험
   - 압력, 온도, 장비 알람, 인터록 등 단순 공정 변수 이상만 있고 실제 누출·노출·폭발·화재 위험의 구체적 서술이 없는 경우
   - 작업중지권을 행사했다는 문구만 있고, 구체적 중대재해 임박 증거가 없는 경우
   - 협력회사 자체 개선완료 건은 대분류나 태그만으로 판단하지 않는다. 작업중지 당시 구체적 물리적 위험이 없거나, 교육·재체결·장비 교체·일정 조율·협의 등으로 즉시 시정 가능한 수준이면 가성으로 판단한다.
   - 구체적 위험어가 있더라도 작업 전 일반 점검에서 발견되었고, 즉시 시정 가능한 수준이면 가성으로 판단한다.
   - 작업 전 계획 단계의 일정 조율, 작업순서 협의, 허가 합의, 단순 작업구역 조정, 즉시 위치 변경으로 해결 가능한 공간·동선·설비 간섭은 가성으로 판단한다. 단, 회사 지정 안전도구·작업방법을 현장 조건상 준수할 수 없어 대체 도구·방법 사용에 대한 담당부서 허락·승인이 필요한 경우는 위 진성 특례를 우선한다.

5. 경계 사례 판단 원칙

   - 위험 유형이 구체적이고, 작업 중 또는 작업 개시 직전이며, 작업을 계속하면 즉시 중대재해 위험에 노출되고, 단순 시정으로 즉시 통제하기 어렵다면 진성으로 판단한다.
   - 위험 유형이 추상적이고, 작업 전 점검 또는 행정·절차 수준이며, 즉시 시정 가능하면 가성으로 판단한다.
   - 제목이나 대분류보다 상세 내용의 구체적 위험 서술을 우선한다.
   - “우려”, “미흡”, “불량”, “위험” 같은 추상어만으로는 진성으로 판단하지 않는다.
   - 추락·감전·질식·화재·폭발·누출·끼임·충돌·붕괴 등 구체적 중대재해 위험이 명시되더라도, 작업 전 일반 점검에서 발견되었고 즉시 시정 가능한 수준이면 가성으로 판단한다.
   - 반대로 작업 중 또는 작업 개시 직전에 위 위험이 현실화되어 작업을 계속할 경우 즉시 중대재해로 이어질 수 있으면 진성으로 판단한다.
   - 물리적 간섭 사례는 “간섭이 있다”는 사실만으로 진성 판단하지 않고, 작업을 계속할 경우 끼임·충돌·추락·설비 파손·누출 등이 임박했는지와 단순 협의·위치 조정으로 즉시 통제 가능한지를 함께 확인한다.
```

## 13. 원하는 답변 형식

가능하면 아래 형식으로 답변해주세요.

```text
1. 핵심 진단
2. 반드시 제거/약화할 문장
3. 추천 패치 문장 3~5개
4. 예상 효과: TP / TR / ACC 방향
5. 코드 후처리 gate가 필요한지 여부
6. 검증 방법
7. 주의해야 할 부작용
```

중요: 원론적인 설명보다 실제 운영 프롬프트에 붙일 수 있는 문장 중심으로 답해주세요.


----답변----
네. 다시 보면 결론을 더 강하게 바꿔야 합니다.

현재 문제는 “프롬프트가 부족한 것”이 아니라, 이미 들어간 가드레일을 다른 진성 특례 문장이 뚫고 있는 구조입니다. 특히 사내 AI가 약한 추론형이라면, 긴 조건문을 안정적으로 따르기보다 배관, 협의, 발판, 추락, DS조치건, 요청완료 같은 단어에 반응할 가능성이 큽니다. 현재 기준선도 TR은 0.82~0.83으로 높지만 TP는 0.60~0.61이고, FP가 327건으로 과다한 상태입니다. 즉 “진성을 놓치지 않는 모델”이 아니라 “진성을 너무 쉽게 주는 모델”입니다.

1. 핵심 진단
1-1. 가장 큰 원인은 작업예정도 진성 가능 특례입니다

현재 프롬프트에서 가장 위험한 문장은 이 계열입니다.

회사 지정 안전기준상 특정 도구·작업방법만 허용되는데 ...
대체 도구·대체 작업방법 사용 전 ...
허락·협의·승인을 받기 위해 작업중지권을 행사한 경우는 ...
진성으로 판단한다.
이는 작업일이 향후 예정되어 있어도 적용한다.

이 문장은 삭제 또는 강한 약화 대상입니다. 이유는 명확합니다. 앞쪽 정의에서는 “작업 개시 직전”을 곧바로 작업에 투입될 상태로 제한하고, 사전점검·계획검토·사전협의는 작업 개시 직전이 아니라고 정의합니다. 그런데 뒤쪽 특례가 “작업일이 향후 예정되어 있어도 적용”한다고 하면서 기본 정의를 우회합니다.

약한 모델 입장에서는 다음처럼 읽힐 가능성이 높습니다.

작업예정 + 공간협소 + 배관 + 협의 + 담당부서 = 진성

반면 실제 목표는 이게 아닙니다.

작업 중/직전 + 안전기준 준수 불가 + 금지설비 밟음 강제성 + 승인/물리대책 없이는 통제 불가 = 진성

즉, 협의가 있었는지가 아니라 협의 없이는 실제 물리적 위험 통제가 불가능했는지가 핵심이어야 합니다.

1-2. 대체 작업방법 승인 = 진성이 너무 독립적인 충분조건처럼 작동합니다

현재 프롬프트는 “부서 협의, 손들기, 승인, 요청완료 문구는 단독으로 진성 근거가 아니다”라고 잘 써놨습니다. 그런데 그 뒤에 “회사 지정 안전도구·작업방법을 사용할 수 없어 대체 방법 승인 필요 시 진성”이라는 특례가 반복됩니다.

문제는 이 특례가 AND 조건처럼 읽혀야 하는데, 실제 LLM은 OR 조건처럼 처리할 수 있다는 점입니다.

원래 의도는 아마 이겁니다.

작업 중/직전
AND 지정 안전방법 사용 불가
AND 승인 없이는 안전기준 위반
AND 물리적 위험이 임박
=> 진성

그런데 현재 문장은 이렇게 오독될 수 있습니다.

작업예정
OR 대체방법 협의
OR 담당부서 승인
OR 공간협소
=> 진성 가능

최근 FP 군집도 정확히 이 방향입니다. pipe/support stepping에서 협력회사 자체 개선완료, 추락/낙하, DS조치건, 공간협소, 협의, 해당없음, 불명확, 아니오가 반복됩니다.

1-3. pipe/support 문장은 방향은 맞지만, 약한 모델에게는 여전히 true-prime입니다

현재 pipe/support 보정 문장 자체는 좋은 시도입니다.

배관·서포트·RACK·덕트 등 ...
① 실제로 밟았거나 밟을 수밖에 없음
② 담당부서 승인이 필요한 대체 작업방법 사용
③ 물리적 대책 필요 또는 수행
중 하나가 직접 확인될 때만 진성

하지만 여기서 중 하나가 위험합니다. 약한 모델은 ② 담당부서 승인이 필요한 대체 작업방법 사용을 넓게 잡아버립니다. 특히 상세 내용에 협의, 승인, 요청완료, DS조치건이 있으면 ②가 충족된 것으로 착각할 수 있습니다.

따라서 pipe/support 계열은 중 하나가 아니라 최소 2개 양성조건 또는 작업 중/직전 + 강제성/물리대책 중 하나로 바꾸는 게 안전합니다.

1-4. 태그 override 금지 원칙이 과하게 절대적입니다

현재 문장은 다음 취지입니다.

대분류·중분류·태그·조치상태·협력회사 자체 개선완료·DS 조치건·조치완료 여부는
상세 내용의 구체적 위험 판단을 뒤집을 수 없다.

이 원칙은 기본적으로 맞습니다. 문제는 최근 FP에서는 협력회사 자체 개선완료, DS 조치건, 삼성전자 임직원 요청완료가 반복적으로 나타나고, 상세 위험이 불명확한데도 위험어만 보고 진성화되는 것입니다. 문서에서도 이 태그들은 자동 가성 조건은 아니지만, 상세 내용이 불명확할 때는 가성 쪽 prior로 써야 할 가능성이 있다고 되어 있습니다.

따라서 이 문장은 이렇게 바뀌어야 합니다.

태그는 명확한 상세 위험을 뒤집지는 못한다.
그러나 상세 위험이 불명확하면 협력회사 자체 개선완료, DS 조치건, 요청완료는 가성 쪽 보조 신호로 사용한다.
2. 반드시 제거/약화할 문장
A. 삭제 1순위

삭제 대상:

이는 작업일이 향후 예정되어 있어도 적용한다.

대체 문장:

작업예정·사전협의·기간연장·허가요청 단계는 원칙적으로 작업 개시 직전으로 보지 않는다. 다만 작업자가 즉시 투입될 예정이고, 승인 또는 물리적 대책 없이는 금지설비 밟음·추락·접액·감전 등 구체적 중대재해 위험에 바로 노출됨이 상세 내용 또는 이미지에서 직접 확인되는 경우에만 예외적으로 진성으로 판단한다.

이렇게 바꾸면 “향후 예정”이 진성 특례로 작동하지 않고, 직접 확인된 즉시 노출 위험이 있을 때만 살아남습니다.

B. 재작성 1순위

기존:

대체 도구·대체 작업방법 사용 전 삼성전자·DS·SEC·담당부서의 허락·협의·승인을 받기 위해 작업중지권을 행사한 경우는 회사 내부 판정 기준상 진성으로 판단한다.

대체:

담당부서 허락·협의·승인은 독립적인 진성 근거가 아니다. 진성 근거로 사용하려면 상세 내용 또는 이미지에서 ① 회사 지정 안전도구·작업방법을 현장 조건상 준수할 수 없음, ② 승인 없이는 작업자가 즉시 구체적 물리 위험에 노출됨, ③ 단순 확인·일정조율이 아닌 안전기준 이탈 승인·위험통제·물리적 보강 협의임이 모두 직접 확인되어야 한다.

핵심은 승인/협의를 독립근거에서 탈락시키는 것입니다. 협의가 있었다가 아니라 협의 목적이 안전통제였고, 그 협의 없이는 즉시 위험이 있었다가 되어야 합니다.

C. 태그 override 문장 재작성

기존:

대분류·중분류·태그·조치상태·협력회사 자체 개선완료·DS 조치건·조치완료 여부는 상세 내용의 구체적 위험 판단을 뒤집을 수 없다.

대체:

대분류·중분류·태그·조치상태는 상세 내용에 직접 확인되는 작업 중 또는 작업 개시 직전의 임박 위험을 뒤집을 수 없다. 그러나 상세 내용에서 작업자 노출, 강제 밟음, 안전기준 이탈 승인 필요성, 물리적 보강 필요성이 불명확하면 협력회사 자체 개선완료·DS 조치건·삼성전자 임직원 요청완료·조치완료는 가성 쪽 보조 신호로 사용한다.

이 문장이 중요합니다. 기존은 태그를 거의 무력화합니다. 새 문장은 명확한 진성은 유지하되, 불명확한 사건에서는 태그를 prior로 살립니다.

D. pipe/support 문장 재작성

기존 pipe/support 문장은 “최소요건”이라고 되어 있지만, 실제로는 ①, ②, ③ 중 하나라서 약한 모델이 ②를 너무 쉽게 만족시킬 수 있습니다.

대체:

배관·서포트·RACK·덕트 등 밟으면 안 되는 설비 관련 사례는 단어가 아니라 직접 확인된 작업상태와 물리적 강제성으로 판단한다. 진성으로 보려면 작업 중 또는 작업 개시 직전임이 확인되고, 실제 밟음·밟을 수밖에 없음·안전발판 부족·지정 안전방법 사용불가·물리적 보강 필요 중 하나 이상이 상세 내용 또는 이미지에서 직접 확인되어야 한다. 협의, 승인, 요청완료, 공간협소, 추락/낙하 대분류, DS조치건, 협력회사 자체 개선완료 문구만 있고 위 물리적 강제성이 불명확하면 가성으로 판단한다.

더 강하게 가려면 이렇게 바꾸는 게 좋습니다.

pipe/support 계열에서 승인·협의만 확인되고 실제 밟음 강제성 또는 물리적 보강 필요성이 확인되지 않으면 진성으로 판단하지 않는다.
3. 추천 패치 문장 3~5개

운영 프롬프트에 바로 붙인다면 아래 5문장만 권합니다. 기존 문장을 계속 늘리는 대신, 최종판정 직전 hard guardrail로 두는 게 낫습니다.

[진성 과판정 방지 패치]

작업예정·사전협의·기간연장·허가요청·요청완료 단계는 원칙적으로 작업 개시 직전으로 보지 않으며, 상세 내용 또는 이미지에서 작업자가 이미 작업 중이거나 즉시 착수 직전임이 직접 확인될 때만 진성 검토 대상으로 올린다.

배관·서포트·RACK·덕트 등 밟으면 안 되는 설비 관련 사례는 협의, 승인, 발판, 공간협소, 추락/낙하, DS조치건, 협력회사 자체 개선완료 문구만으로 진성 판단하지 않는다.

해당 계열은 실제 밟음, 밟을 수밖에 없음, 안전발판 부족, 지정 안전방법 사용불가, 물리적 보강 필요 또는 실시 중 적어도 하나가 상세 내용 또는 이미지에서 직접 확인되고, 그 위험이 작업 중 또는 작업 개시 직전의 작업자 노출과 연결될 때만 진성으로 판단한다.

forced_stepping_level, standard_method_blocked, approval_purpose, physical_countermeasure, 작업자 위험 노출이 해당없음·불명확·아니오·없음이면 그 항목은 진성 근거가 아니라 가성 보정 신호로 처리한다.

협력회사 자체 개선완료·DS 조치건·삼성전자 임직원 요청완료·조치완료는 자동 가성 사유는 아니지만, 상세 내용에서 작업 중 또는 작업 개시 직전의 임박한 물리적 위험이 직접 확인되지 않으면 가성 쪽 보조 신호로 사용한다.

개인적으로는 이 5문장 중에서도 핵심은 1, 3, 4번입니다. 2번은 trigger token 억제용이고, 5번은 태그 prior 복원용입니다.

4. 예상 효과: TP / TR / ACC 방향

현재 대략값은 다음과 같습니다.

전체 약 3,142건
ACC 약 0.86
TR 약 0.82~0.83
TP 약 0.60~0.61
FN 104건
FP 327건

이 수치라면 실제 진성 수는 대략 590건 전후로 추정됩니다. FN이 104건이고 TR이 0.82~0.83이면, 실제 진성 총량은 약 580~610건 수준입니다. 그러면 현재 진성으로 예측한 건 중 실제 진성은 약 480~500건, FP는 327건 정도로 볼 수 있습니다.

TP를 0.70 이상으로 올리려면, 현재 TP count가 그대로 유지된다고 가정할 때 FP를 대략 327건 → 205~215건 수준까지 줄여야 합니다. 즉 FP를 최소 110~125건 정도 줄여야 합니다.

ACC 0.88은 더 쉽습니다. 현재 오류는 FN 104 + FP 327 = 431건입니다. 3,142건 기준 ACC 0.88을 넘기려면 오류가 약 377건 이하가 되어야 하므로, 순오류를 54건 이상만 줄이면 됩니다. 즉 gate가 FP를 100건 줄이고 새 FN을 40건 만들더라도 ACC는 개선됩니다.

예상 방향은 이렇습니다.

Prompt patch only:
- TP: 0.61 → 0.65~0.68 가능
- TR: 0.82 → 0.78~0.81 정도로 소폭 하락 가능
- ACC: 0.86 → 0.87 전후

Prompt patch + deterministic gate:
- TP: 0.61 → 0.70 이상 가능
- TR: 0.82 → 0.72~0.78 범위로 하락 가능
- ACC: 0.88 이상 가능

단, gate가 너무 강하면 TR이 0.70 아래로 내려갈 수 있습니다. 그래서 “작업예정이면 무조건 가성” 같은 gate는 금지해야 합니다. gate는 작업예정 + 불명확 evidence + 물리대책 없음 + 행정협의 신호가 동시에 있을 때만 걸어야 합니다.

5. 코드 후처리 gate가 필요한지 여부

필요합니다. 이 케이스는 프롬프트만으로 해결하기 어렵습니다.

이유는 세 가지입니다.

첫째, 이미 프롬프트 안에 “협의 단독 근거 아님”, “pipe/support 최소요건 확인”, “해당없음/불명확/아니오는 근거 부족 신호”가 들어가 있습니다. 그런데도 FP가 반복됩니다. 즉 문장이 없어서가 아니라, 모델이 그 문장을 안정적으로 집행하지 못하는 것입니다.

둘째, evidence schema는 이미 gate에 쓸 수 있는 구조화 필드를 갖고 있습니다. forced_stepping_level, standard_method_blocked, approval_purpose, physical_countermeasure, admin_only_signal, pipe_true_discriminator, pipe_false_guardrail 등이 이미 있습니다.

셋째, 사내 AI가 약한 추론형이면 긴 프롬프트보다 deterministic rule이 재현성이 높습니다. 특히 true_confirm을 이미 쓰고 있는데도 FP가 327건이면, 같은 LLM을 두 번 돌리는 방식만으로는 부족합니다.

추천 gate는 아래처럼 잡는 게 안전합니다.

PIPE_TRUE_PURPOSES = {

    "위험통제협의",
    "물리적보강협의",
    "대체작업방법승인",
}

PIPE_ADMIN_PURPOSES = {
    "단순이메일협의",
    "일정조율",
    "일반허가",
    "단순공유",
    "해당없음",
    "불명확",
}

PIPE_POSITIVE_FORCE = {
    "실제밟음",
    "밟을수밖에없음",
    "안전발판부족",
}

PIPE_WEAK_FORCE = {
    "작업방법상가능성",
    "단순검토",
    "없음",
    "불명확",
}

PIPE_POSITIVE_COUNTERMEASURES = {
    "보강판",
    "비계",
    "임시발판",
    "난간",
    "덮개",
    "작업방법변경",
    "설비보호조치",
}

PIPE_PREWORK_PHASES = {
    "작업전계획",
    "사전협의",
    "작업예정",
    "기간연장",

PIPE_ACTIVE_PHASES = {
    "작업중",

}

def pipe_support_false_gate(evidence: dict) -> tuple[bool, str]:
    p = evidence.get("pipe_support_evidence", {})

    if p.get("is_pipe_support_case") != "예":
        return False, "not_pipe_support"

    phase = p.get("work_phase_for_pipe")
    actual_work = p.get("actual_or_planned_work")
    forced = p.get("forced_stepping_level")
    standard_blocked = p.get("standard_method_blocked")
    permission_required = p.get("permission_required_by_company_rule")
    approval_purpose = p.get("approval_purpose")
    admin_only = p.get("admin_only_signal")
    countermeasures = set(p.get("physical_countermeasure") or [])
    physical_action_status = p.get("physical_action_status")

    phase_positive = (
        phase in PIPE_ACTIVE_PHASES
        or actual_work in {"실제작업중", "작업개시직전위험구역진입"}
    )

    force_positive = forced in PIPE_POSITIVE_FORCE

    approval_positive = (
        standard_blocked == "예"
        and permission_required == "예"
        and approval_purpose in PIPE_TRUE_PURPOSES
    )

    countermeasure_positive = (
        bool(countermeasures & PIPE_POSITIVE_COUNTERMEASURES)
        and physical_action_status in {"실시됨", "실시필요"}
    )

    admin_negative = (
        phase in PIPE_PREWORK_PHASES
        or actual_work in {"작업전현장확인", "작업예정문서협의", "기간연장요청", "불명확"}
        or forced in PIPE_WEAK_FORCE
        or approval_purpose in PIPE_ADMIN_PURPOSES
        or admin_only == "예"
        or countermeasures <= {"없음", "불명확"}
    )

    # 강한 진성 예외: 작업 중/직전 + 물리적 강제성 또는 물리대책
    strong_true = (
        phase_positive
        and (force_positive or countermeasure_positive or approval_positive)
    )

    if strong_true:
        return False, "strong_pipe_true_evidence"

    # 강한 가성 gate: 사전/행정 신호가 있고, 진성 양성조건이 없음
    positive_count = sum([
        phase_positive,
        force_positive,
        approval_positive,
        countermeasure_positive,
    ])

    if admin_negative and positive_count == 0:
        return True, "pipe_admin_or_unclear_no_positive_evidence"

    # 중간 gate: 진성 양성조건이 1개뿐이고, 그 1개가 승인/협의뿐이면 가성 보정
    if positive_count == 1 and approval_positive and not phase_positive and not force_positive and not countermeasure_positive:
        return True, "approval_only_without_phase_force_countermeasure"

    return False, "no_gate"

운영에서는 이렇게 쓰면 됩니다.

llm_result = final_judge_result

gate_hit, gate_reason = pipe_support_false_gate(evidence)

if llm_result["판정"] == "진성" and gate_hit:
    llm_result["판정"] = "가성"
    llm_result["review_needed"] = True
    llm_result["applied_step"] = "가성조건"
    llm_result["판단근거"] = (
        "pipe/support 계열이나 작업 중/작업개시직전의 실제 밟음 강제성, "
        "안전기준 이탈 승인 필요성, 물리적 보강 필요성이 직접 확인되지 않아 가성으로 보정함."
    )
    llm_result["gate_reason"] = gate_reason

여기서 중요한 점은 pipe_true_discriminator와 pipe_false_guardrail을 그대로 믿지 않는 것입니다. 이 둘은 LLM이 만든 파생 필드라서 오염될 수 있습니다. gate는 forced_stepping_level, work_phase_for_pipe, approval_purpose, standard_method_blocked, physical_countermeasure 같은 primitive field를 기준으로 다시 계산하는 게 낫습니다.

6. 검증 방법

검증은 세 벌로 나눠야 합니다.

A. current_policy baseline
B. prompt patch only
C. prompt patch + deterministic gate

성공 기준은 단순 ACC 하나로 보면 안 됩니다. 최소한 아래를 봐야 합니다.

전체 지표:
- ACC
- TP, 즉 AI가 진성이라고 한 건 중 실제 진성 비율
- TR, 즉 실제 진성을 진성으로 잡은 비율
- FP count
- FN count
- AI 진성 예측 비율

핵심 군집 지표:
- pipe/support stepping FP count
- pipe/support stepping TP
- pipe/support stepping TR
- 협력회사 자체 개선완료 내 FP count
- DS 조치건/요청완료 내 FP count
- gate가 진성→가성으로 뒤집은 건수
- gate가 뒤집은 건 중 실제 가성 비율

특히 gate 검증에서는 이 지표가 가장 중요합니다.

gate_flip_precision = gate가 진성→가성으로 바꾼 건 중 실제 가성 비율

이 값이 0.70 미만이면 gate가 너무 거칩니다. 0.75 이상이면 운영 투입 가치가 있습니다. 0.80 이상이면 매우 좋은 gate입니다.

권장 성공 기준은 아래처럼 잡겠습니다.

1차 성공:
- TP >= 0.68
- ACC >= 0.875
- TR >= 0.74
- FP 327건 대비 최소 80건 이상 감소

운영 성공:
- TP >= 0.70
- ACC >= 0.88
- TR >= 0.70
- FP 327건 대비 최소 110건 이상 감소

강한 성공:
- TP >= 0.72
- ACC >= 0.89
- TR >= 0.72
- pipe/support FP 40% 이상 감소

검증 샘플은 2,000~3,000건이면 충분합니다. 다만 split은 랜덤만 쓰면 안 됩니다. 반드시 아래 stratified slice를 따로 봐야 합니다.

- pipe/support stepping
- space/path/equipment interference
- contractor self-correction
- 협력회사 자체 개선완료
- DS 조치건 / 삼성전자 임직원 요청완료
- 추락/낙하 대분류
- 미상 액체 / 냄새 / 가스 / 누출

이 중 미상 액체 / 냄새 / 가스 / 누출은 pipe gate에서 제외하는 게 안전합니다. 해당 계열은 작업중지 당시 위험성을 배제할 수 없으면 진성 가능성을 우선 검토하는 별도 로직이 있고, pipe/support FP와 성격이 다릅니다.

7. 주의해야 할 부작용
부작용 1. 실제 진성인 사전승인형 사건을 가성으로 떨어뜨릴 수 있음

예를 들어 작업은 아직 시작 전이지만, 현장 구조상 지정 사다리나 발판을 사용할 수 없고, 작업자가 투입되면 케미컬 배관을 밟을 수밖에 없으며, 담당부서 승인 없이는 안전기준 준수가 불가능한 사건이 있을 수 있습니다. 이런 사건은 진성으로 유지해야 합니다.

그래서 gate는 작업예정 하나만으로 가성화하면 안 됩니다. 반드시 아래 조합일 때만 가성화해야 합니다.

작업예정/사전협의/기간연장
+ 실제 밟음 강제성 없음/불명확
+ 물리적 보강 없음/불명확
+ approval_purpose가 단순협의/일정조율/일반허가/불명확
부작용 2. evidence extraction 오류가 final gate 오류로 증폭될 수 있음

현재 evidence 단계에서 해당없음, 불명확, 아니오가 많이 나오는 것은 좋은 신호일 수 있지만, evidence extractor가 실제 위험을 놓친 경우에는 gate가 오판을 고정할 수 있습니다. 따라서 gate 적용 후 review_needed=true를 남기는 게 좋습니다.

운영 DB에는 아래 필드를 추가하는 걸 권합니다.

original_llm_label
final_label_after_gate
gate_applied
gate_reason
positive_pipe_signals
negative_pipe_signals
review_needed

이렇게 남기면 나중에 담당자가 “왜 진성에서 가성으로 바뀌었는지”를 추적할 수 있습니다.

부작용 3. 프롬프트만 바꾸면 재현성이 낮을 수 있음

사내 AI가 약한 추론형이면, “단어 자체를 보지 말고 물리적 강제성을 봐라” 같은 문장을 매번 안정적으로 지키지 못할 수 있습니다. 이미 evidence prompt에는 “협의라는 단어 자체를 핵심 근거로 쓰지 말라”, “실제 밟음 여부와 물리적 대책 필요성을 구분하라”는 지시가 들어가 있습니다. 그런데도 final judge가 진성으로 올라가는 문제가 보고되어 있습니다.

따라서 최종 추천은 이겁니다.

1. `작업일이 향후 예정되어 있어도 적용한다` 삭제
2. `대체 작업방법 승인` 특례를 독립 진성 근거가 아닌 AND 조건으로 재작성
3. 태그 override 금지 원칙을 “명확한 위험일 때만 태그보다 우선”으로 약화
4. pipe/support 계열에 deterministic false gate 추가
5. gate가 뒤집은 건은 review_needed=true로 남기기

최종적으로는 prompt patch only보다 prompt patch + code gate가 맞습니다. 지금 목표가 TP 0.70 이상이고 FP가 327건이면, 프롬프트 문장 몇 개로 충분히 줄이기 어렵습니다. deterministic gate로 최소 110건 안팎의 FP를 줄이는 방향이 현실적입니다.
